# Phase 3: Clarifying Questions - 澄清与决策

## 1. 首版范围

**问题**：首版是否只做「首页 + 对话」最小闭环，其余场景（托管日历、门店经营、巡检、物流、市场、合同等）是否作为静态占位页？

**决策**：✅ **是**

- 先实现「首页（操作指引）+ 侧栏 + 对话（新建/列表/流式）」闭环
- 托管日历、门店经营、巡检、物流、市场、合同等先做**静态占位页**（与参考页布局一致，无真实数据）
- 后续再接 Mock 或真实接口

**理由**：便于先跑通前后端与对话流，降低首版复杂度。

## 2. Mock 工具粒度

**问题**：Mock 工具的数量与粒度？

**决策**：✅ **提供 2 个 Mock 工具**

- `get_store_summary(storeCode?: string)`：门店经营摘要
  - 返回：销售额、客流、待办数量等
  - 数据：写死或简单随机

- `get_finance_diff(month?: string)`：销售凭证与金蝶差异
  - 返回：差异列表示例
  - 数据：写死示例数据

**理由**：工具描述与返回结构贴近「门店经营」「金蝶凭证差异」等页面语义，便于后续替换为真实 MCP。

## 3. 可观测性

**问题**：首版是否集成 Langfuse/OpenTelemetry？

**决策**：⏸️ **可选**

- 在架构文档中预留接口
- 实现阶段再决定是否接入
- 首版可暂不接入，后续根据需求添加

## 4. 其他决策

### 4.1 会话存储

- **首版**：前端本地存储（localStorage）
- **后续**：可接入后端会话管理API

### 4.2 错误处理

- 统一错误体格式：`{ code, message, details? }`
- 前端统一错误提示组件

### 4.3 开发环境

- 前端：Vite dev server（代理到后端）
- 后端：Spring Boot dev server
- 本地联调：前后端分别启动
